# Lumina Language Guide

## Overview

Lumina is an Actor Model architecture definition language. `.lm` files describe *what* a system does — types, actors, interfaces, and intent. They contain **no control flow** and **no implementation logic**. Implementation is generated by AI agents based on the contracts defined in the source.

## File Structure

A `.lm` file contains three kinds of top-level declarations:

| Declaration | Purpose |
|-------------|---------|
| `import "path" as alias` | Import types and modules from another file |
| `type Name = ...("desc")` | Define a reusable data structure with semantic description |
| `module Name { ... }` | Define an actor with its interface and intent |

## Types

### Primitives

```
Int      — integer
Float    — floating point number
String   — text
Bool     — boolean
```

Each primitive can carry a semantic description:

```rust
Int("unique user identifier")
Float("transaction amount in USD")
String("sanitized display name")
```

### Records (structs)

```rust
type User = {
    "id": Int("database primary key"),
    "name": String("display name"),
    "email": String("contact email")
}("user account record")
```

### Generics

```rust
List<Int>("ordered list of scores")
Map<String, Float>("product name to price mapping")
```

### Unions

```rust
type Result = {
    "ok": Int
} | {
    "err": String
}
```

### Qualified Types (cross-file references)

```rust
import "types.lm" as t

// t.DataPacket refers to DataPacket defined in types.lm
type EnrichedPacket = {
    "data": t.DataPacket,
    "timestamp": Int("unix epoch ms")
}
```

## Modules

A module is an actor — the fundamental unit of computation in Lumina. Every module has:

| Section | Required | Purpose |
|---------|----------|---------|
| `setup` | No | Natural-language description of initialization |
| `interface` | Recommended | JSON message contract: method name → input/output schemas |
| `logic` | Recommended | Natural-language description of what the module does |
| `actor` | No | Mount child modules as sub-actors |

```rust
module Sorter {
    setup: "allocate a contiguous memory pool for high-frequency sorting"

    interface: {
        "do_sort": { "raw": t.DataPacket } -> { "sorted": t.DataPacket }
    }

    logic: "implement a non-recursive quicksort on the payload array"
}
```

### The Interface Contract

Each method in the interface defines:

- **Input**: a record type — what JSON fields the method expects
- **Output**: a record type — what JSON fields the method returns

```rust
interface: {
    "add": { "a": Int, "b": Int } -> { "result": Int }
    "status": { } -> { "uptime": Float, "requests": Int }
}
```

Methods with empty input `{ }` take no parameters. The output is always a JSON object.

### Child Actors

Modules can mount other modules as child actors:

```rust
module Controller {
    actor sorter: worker.Sorter
    actor cache: db.Cache

    logic: "receive data → sorter.do_sort → cache.store"
}
```

The parent communicates with children via `send_message(actor_name, method, params)`. The runtime handles routing.

### Modules Without Interface

A module with only `setup` and `logic` (no `interface`) is a **spontaneous actor** — it initiates messages but doesn't receive external calls:

```rust
module CronGenerator {
    setup: "start a timer that fires every 5 seconds"

    logic: "generate a random DataPacket and send it to downstream actors"
}
```

## Imports

Imports make types and modules from other files available:

```rust
import "types.lm" as t
import "db/postgres.lm" as db
```

The alias is a local namespace. Reference imported types with `alias.TypeName` and imported modules with `alias.ModuleName`.

## Design Conventions

### Semantic Strings Are First-Class

Every `("description")` in a type definition is **not a comment**. It's metadata that gets compiled into the Task JSON and passed to the AI agent as implementation context. Write meaningful descriptions — the AI uses them.

Good:
```rust
Int("user age in years, must be >= 0")
```

Bad:
```rust
Int("an integer")
```

### Module Granularity

A module should do **one thing well**. If your `logic` description starts listing unrelated tasks, split it into multiple modules.

### Interface Completeness

Define every method the module responds to. The interface is the contract — the AI agent implements exactly those methods. If a method isn't listed, it won't be implemented.

## Configuration

### Lumina.toml

```toml
[project]
name = "my-project"
language = "python"      # target language for generated code

[build]
mode = "monolith"        # "monolith" or "microservice"

# Enable auto-testing for specific modules (requires Claude Code agent)
[modules.CriticalModule]
test = true
```

### Agent Configuration

Lumina supports two AI backends for code generation. Switch with `--agent`:

```bash
lumina build --agent llm           # Pure LLM (default)
lumina build --agent claude_code   # Claude Code CLI
```

**LLM agent** (`--agent llm`):
- Requires `OPENAI_API_KEY` environment variable
- Optional: `OPENAI_API_BASE` (default: https://api.openai.com/v1), `OPENAI_MODEL` (default: gpt-4o)
- Code generation only — no testing capability

```bash
export OPENAI_API_KEY=sk-xxx
lumina build --agent llm
```

**Claude Code agent** (`--agent claude_code`):
- Requires `claude` CLI installed and in PATH
- Install: `npm install -g @anthropic-ai/claude-code`
- On Windows, also needs git-bash in PATH

```bash
npm install -g @anthropic-ai/claude-code
lumina build --agent claude_code
```

Optional env vars for non-standard setups:
```bash
CLAUDE_CODE_PATH=/path/to/claude       # if not in PATH
CLAUDE_CODE_GIT_BASH_PATH=/path/to/bash  # Windows git-bash location
```

## CLI Reference

```bash
lumina init <name>                 # Create a new project
lumina build                       # Build with default agent (llm)
lumina build --agent claude_code   # Build with Claude Code agent
lumina build --dry-run             # Preview Task JSONs without calling AI
lumina build --mode microservice   # Override build mode
lumina parse <file.lm>             # Show parsed AST
lumina parse <file.lm> -f json     # Show Task JSON
```
